## 6. Invitations RLS fix

**Bug:** Admin invites a rep → pending row appears momentarily → reload → pending list is empty. The rep eventually accepts and shows as Active, but during the "pending" window they're invisible.

**Root cause:** The `invitations__select__admin_or_token` RLS policy was:

```sql
USING (
  is_admin(workspace_id)
  OR email = ((SELECT users.email FROM auth.users WHERE users.id = auth.uid()))::text
)
```

The OR subquery reads `auth.users`. The `authenticated` role doesn't have `SELECT` on `auth.users`, so the whole policy errored and PostgREST silently denied all SELECTs.

**Fix:** wrapped the auth.users lookup in a `SECURITY DEFINER` helper.

```sql
CREATE OR REPLACE FUNCTION public.current_user_email() RETURNS text ... SECURITY DEFINER
AS $$ SELECT email::text FROM auth.users WHERE id = auth.uid(); $$;

DROP POLICY ...; CREATE POLICY ... USING (is_admin(workspace_id) OR email = current_user_email());
```

**Migration:** `supabase/migrations/20260511000001_fix_invitations_rls.sql` (applied to production already).

**Commit:** `0365185`

## 10. Analytics 1000-row cap — real fix

**First attempt (wrong, in `f4e8e99`):** added `.range(0, 99999)` to analytics routes that fetched from `leads`/`emails`/`call_logs`. **This does not work.** PostgREST has a server-side `db-max-rows` cap (1000) that `.range()` and `.limit()` cannot bypass. Verified via curl.

**Real fix (in `0011830`):** single-row jsonb responses are exempt from the row cap. New SQL function:

```sql
-- supabase/migrations/20260512000001_analytics_batch_stats_rpc.sql
CREATE FUNCTION public.get_batch_analytics(p_workspace_id uuid, p_batch_ids uuid[])
RETURNS jsonb LANGUAGE sql STABLE SECURITY DEFINER ...
```

Returns `{leads: [...], emails: [...]}` with per-batch aggregates computed in SQL. `app/api/analytics/batches/route.ts` calls the RPC instead of fetching rows.

**Side-effect fix:** the previous route queried `emails.batch_id` — a column that doesn't exist on the emails table. Email-per-batch stats had always been silently wrong. The RPC derives batch_id via `emails → leads.batch_id` join.

**Verified:** batch with 3,008 leads now reports 3,008.

**Open issue (see §14):** `time-series`, `email-metrics`, `funnel`, `reps` analytics routes still have the `.range(0, 99999)` from the first attempt. They'll silently report capped numbers when those tables exceed 1000 rows. Each needs its own SQL aggregate RPC.

**Commits:** `f4e8e99` (band-aid), `0011830` (real fix for batches)

## 13. Status revert bug (double-PATCH)

**Symptom user reported:** Clicking "Bad Lead" in the side panel on certain leads (Roderick Leonard, Kris Kollevoll) → status flips back to "New" instead of staying as "Bad Lead". Reproducible.

**Root cause:** Two PATCHes fired for every status change made from the side panel inside the leads list page.

1. **Side panel** (`components/leads/lead-full-panel.tsx`) optimistically updates its local state, then PATCHes `/api/leads/:id`.
2. **Side panel** calls `onLeadChange({ status })` to notify the parent list.
3. **Parent** (`app/(dashboard)/leads/leads-client.tsx`) called `handleStatusChange(...)` which fired a second PATCH.

Confirmed in the DB: every status change wrote two duplicate `lead_status_changed` activity rows milliseconds apart. On a race, the parent's `catch` block rolled back the entire local list to its pre-click state — visible to the user as "the status reverted."

**Fix (in `ff5606a`):** parent's `onLeadChange` just updates the local list row now, no second PATCH:

```ts
onLeadChange={(patch) => {
  setLeads((prev) => prev.map((l) =>
    l.id === selectedLeadId ? { ...l, ...patch, updated_at: new Date().toISOString() } : l
  ))
}}
```

**Detail page (`/leads/[id]`) is unaffected** — `lead-detail-client.tsx` has its own `handleStatusChange` and doesn't double-fire.

**Pipeline page is also unaffected** — `patchLead` was already a pure local update, no API call.

**Commit:** `ff5606a`

## 15. Quirks worth remembering

### PostgREST `db-max-rows` is hard-capped at 1000

`.range()` and `.limit()` do NOT bypass it. The only way to fetch >1000 rows is:

- A SQL function returning a single jsonb value (single-row responses are exempt)
- A specific RPC like `get_workspace_leads_json` that's already designed around this

If you see a query that does `.from('leads').select(...)` without an explicit limit *and* needs to aggregate, it's silently wrong above 1000 rows.

### Schema drift — repo migrations vs deployed DB

`ai_usage_logs.cached` is the known case. There may be others. Before adding columns to inserts based on what the repo migration says, verify the column exists in the live DB.

### Gmail compose URL is plain text only

No HTML. No markdown. The Unicode bold trick in `styleSnapshotBody()` works for visual emphasis. If you need real rich text, you have to use the Gmail API (OAuth) — not the URL.

When ambiguous, the safer bet is to ask. The user re-enabled auto a few times after exiting it.
